## 2024-05-18 - Prevent Path Traversal in Flat-File Storage
**Vulnerability:** The application accesses tasks, knowledge base files, and logs dynamically across `app.py`, `src/satya/core/storage.py`, and `src/satya/sdk/client.py`. These components constructed file paths via `os.path.join()` using unvalidated strings (like `filename`, `agent_name`, or `task_id`) directly, which allowed malicious actors or malfunctioning agents to traverse directories (e.g., passing `../../etc/passwd` to delete or read unauthorized files).
**Learning:** In a zero-database (flat-file) architecture, all IDs and names directly translate to file system paths. Because these flat directories are used exclusively without deep nesting, we can securely isolate file access using `os.path.basename()`. Additionally, we learned that the `shell=True` parameter in `src/satya/core/completion.py` is intentionally kept to support command shell operators in test tasks, and replacing it with `shell=False` causes test regressions.
**Prevention:** Always sanitize any filename or task ID with `os.path.basename()` before passing it to `os.path.join()` when interacting with storage directories (`TRUTH_DIR`, `TASKS_DIR`, `AGENTS_DIR`).
