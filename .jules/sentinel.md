## 2024-05-18 - Prevent Path Traversal in Flat-File Storage
**Vulnerability:** The application accesses tasks, knowledge base files, and logs dynamically across `app.py`, `src/satya/core/storage.py`, and `src/satya/sdk/client.py`. These components constructed file paths via `os.path.join()` using unvalidated strings (like `filename`, `agent_name`, or `task_id`) directly, which allowed malicious actors or malfunctioning agents to traverse directories (e.g., passing `../../etc/passwd` to delete or read unauthorized files).
**Learning:** In a zero-database (flat-file) architecture, all IDs and names directly translate to file system paths. Because these flat directories are used exclusively without deep nesting, we can securely isolate file access using `os.path.basename()`. Additionally, we learned that the `shell=True` parameter in `src/satya/core/completion.py` is intentionally kept to support command shell operators in test tasks, and replacing it with `shell=False` causes test regressions.
**Prevention:** Always sanitize any filename or task ID with `os.path.basename()` before passing it to `os.path.join()` when interacting with storage directories (`TRUTH_DIR`, `TASKS_DIR`, `AGENTS_DIR`).
## 2024-05-18 - XSS and DoS Prevention in Streamlit Output Rendering
**Vulnerability:** `format_date` and `format_time_ago` functions in `app.py` were returning unescaped original strings in their exception handling blocks when parsing invalid ISO timestamp strings. Similarly, the activity log rendering crashed when attempting to parse invalid ISO timestamp strings. These unescaped values were subsequently injected into HTML components with `unsafe_allow_html=True` using `st.markdown`, posing Cross-Site Scripting (XSS) risks. The unhandled exception in activity log rendering also resulted in a Denial of Service (DoS) where the entire Streamlit page crashed and could not be rendered.
**Learning:** Any user-provided data output implicitly or explicitly to a frontend web page must be sanitized, especially when dealing with potential failure paths like parsing invalid input strings. The `st.markdown(..., unsafe_allow_html=True)` acts as a direct sink for unsanitized data, so all data flowing into it must be verified to be HTML escaped. Also, failure to parse strings in rendering loops can lead to fatal exceptions, crashing the dashboard entirely.
**Prevention:** All formatted outputs from date utilities that might fail to parse user data should fallback to `html.escape(str(invalid_input))`. Critical rendering blocks processing potentially malformed data (like timestamps from external agents) should use try-except blocks, falling back to an escaped string, to guarantee graceful and secure degradation instead of an unhandled `ValueError`.

## 2026-02-07 - SSRF Protection with Manual Redirect Validation
**Vulnerability:** The `Scraper.fetch_and_save` function used `requests.get` with default redirect handling, allowing access to internal services or metadata endpoints (SSRF).
**Learning:** Standard library `socket.getaddrinfo` should be used to resolve all IPs for a hostname to prevent bypasses. `requests` automatic redirect handling must be disabled and implemented manually to validate each hop in the redirect chain.
**Prevention:** Implement a robust `_is_safe_url` helper and manual redirect loop for any component that fetches external URLs. Always resolve hostnames to all associated IP addresses (IPv4 and IPv6) to prevent DNS rebinding attacks and ensure all destinations are safe.
